结论先说：
Advoo 重连中 只会在“进入某个已有会话后，前端判断这个会话的上一次运行还没正常结束，于是发起恢复订阅（reconnect）”时出现。

**1. 代码里的硬触发条件（100%）**

1. 触发历史加载 loadHistoryMessages(sessionId)（进入已有会话时）。
2. 历史事件 events.length > 0。
3. 历史最后一条事件 lastEventType !== RUN_END && lastEventType !== RUN_ERROR。
4. 进入 reconnect 状态：resume_reconnect + set_reconnecting(true)。
5. UI 将 statusVariant 渲染为 reconnecting，显示文案 chat.runtimeStatus.reconnecting。

**2. 导致第 3 条成立的“所有末事件类型”（穷举）**
除了 RUN_END、RUN_ERROR 外，末事件只要是以下任意一种，都会触发“重连中”：

- RUN_START
- RUN_CANCEL
- LLM_CALL_START
- LLM_CALL_END
- TEXT_START
- TEXT_DELTA
- TEXT_END
- TEXT
- TOOL_CALL_START
- TOOL_CALL
- TOOL_CALL_DELTA
- TOOL_CALL_END
- TOOL_RESULT
- 以及“事件类型缺失/异常值”（代码里也会被当作“未结束”）

**3. 对应到业务场景的全量原因（推断自代码）**

- 刷新页面/重新进入会话时，该 run 仍在执行。
- 切到一个仍在执行中的历史会话。
- 取消流程中只落了 RUN_CANCEL，尚未到 RUN_END。
- SSE 中断、页面离开、网络波动，导致你离开时 run 还没写入终态事件。
- 后端事件持久化延迟，RUN_END/RUN_ERROR 还没进历史。
- 后端异常或事件丢失，终态事件缺失。
- 历史事件异常（末事件类型缺失/不合法）。

**4. 为什么有时会“重连中”持续较久**

- subscribe_stream 本身有重试（最多 3 次，指数退避），握手期间会一直处于 reconnecting。
- 代码只有在“收到第一条恢复后的 live 事件”或“流关闭收尾”后才退出 reconnecting。

**5. 明确不会出现“重连中”的场景**

- 历史事件为空。
- 历史末事件是 RUN_END 或 RUN_ERROR。
- 新建会话首次发送（走 execute，不走 history reconnect）。
- isLoadingHistory 阶段（此时显示的是“加载历史”页面，不是重连条）。
- cancelling 优先级高于 reconnecting（会显示取消中）。
