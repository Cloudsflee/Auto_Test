# 压缩测试机制现状改造完整方案（按问题导向重构版）

## 1. 整合你之前提到过的问题（逐条归纳）

以下问题都来自你前面的连续反馈，我这里做一次结构化汇总：

1. 预埋信息点设计不合理。
  - 你指出：像活动编号这类“高优先级硬信息”几乎不丢，测不出压缩边界。
  - 你也指出：若信息天然会进记忆文件（`workbook.md/files.md`），在记忆机制正常时也难丢。
2. 当前“丢失判定”与“业务偏航”混淆。
  - 会话里经常出现任务重心切换，模型不沿用旧锚点，不一定是忘记，可能只是执行当前任务。
  - 如果判定规则不区分，会把“换题/改口径”误报成“压缩遗忘”。
3. 结果对“压缩因果”不够强。
  - 目前更像“某轮答不出来了”，但不能严谨回答“是不是压缩导致的”。
  - 缺少压缩事件与探针判定的时间对齐与归因。
4. 统计口径不够能指导迭代。
  - 只看“第几轮失败”价值有限。
  - 需要能区分：可接受裁剪 vs 不可接受丢失；压缩前已有问题 vs 压缩后新问题。
5. 与现有框架结合要现实可落地。
  - 你希望不是推倒重来，而是能在现有 `auto_test`（已有 session 输出、LLM 双层判定、results 结构）上演进。

## 2. 根据问题，可尝试的解决方式（对应上面每条）

## 2.1 针对“预埋信息点不合理”：从二分法改为“保留契约分层”

不要只分“重要/不重要”，改为三层：

1. `must_keep`（必须保留）
  - 丢失即失败（红线）。
2. `should_keep`（应当保留）
  - 压缩评估主战场：允许偶发波动，不允许稳定退化。
3. `may_drop`（可裁剪）
  - 丢失不计失败，用于验证压缩是否有效减负。

这样可以避免：

1. 全测“硬信息”导致“永远不丢”的假阳性稳定。
2. 全测“软信息”导致“丢了也正常”的假阴性宽松。

## 2.2 针对“判定混淆”：保留双层 LLM，但引入 cause 语义

将当前 `suspect + verify` 扩展为：

1. 判定三态：`remembered / ambiguous / forgotten`
2. 必填归因：`compression_related / task_switch / instruction_override / insufficient_evidence`

计入压缩失败的严格条件：

1. `forgotten`
2. 且 `cause=compression_related`
3. 且满足连续失败阈值

这样可以把“业务换题”从“压缩故障”中剥离。

## 2.3 针对“压缩因果弱”：引入压缩时点对齐

每个 session 增加一条 `compaction_timeline`：

1. 捕捉“记忆压缩完成/会话记忆压缩完成”等信号对应 turn。
2. 所有探针判定标记 `pre_compaction` 或 `post_compaction`。

只把 `post_compaction` 新增失败视为压缩相关候选问题。

## 2.4 针对“统计不可用”：从“轮次失败”升级为“分层保留统计”

核心统计不再只看 first failure turn，而是看：

1. `retention_rate_by_tier`
2. `post_compaction_delta_by_tier`（压缩后新增失败率）
3. `false_alarm_rate`（被归因为 task_switch/override 的比例）
4. `first_post_compaction_failure_turn` 分布

这样报告才对“压缩策略迭代”有指导价值。

## 2.5 针对“要兼容现有框架”：先增量改，不重构主干

现有框架可直接复用：

1. `run_memory_compression_failure_scan.py`（主扫描流程）
2. `memory_failure_llm.py`（探针问句与双层判定）
3. 现有 `results/memory_failure/<run>/...` 目录

改法是“补字段 + 补文件 + 补统计”，不是重写框架。

## 3. 如何与当前项目自动化测试框架结合，统计怎么做更合理

下面是直接按当前代码形态可落地的结合方案。

## 3.1 流程结合（不推翻现有 scan）

在现有 session 循环中增加阶段标签：

1. `stage_0_reset`
2. `stage_1_plant`
3. `stage_2_workload`
4. `stage_3_probe`
5. `stage_4_post_compaction_probe`

并在每次 probe 结果中落：

1. `retention_tier`
2. `judge_state`
3. `cause_hint`
4. `compaction_phase`

## 3.2 数据结合（新增但兼容）

保留现有：

1. `run_data/probe_checks.json`
2. `run_data/turn_results.json`
3. `session_meta.json`

新增：

1. `run_data/probe_timeline.json`
2. `run_data/compaction_events.json`
3. `aggregate/compression_effect_summary.json`
4. `aggregate/compression_effect_report.md`

保证老脚本/老结果读取不被破坏。

## 3.3 探针资产结合（用 dataset 管而不是硬编码）

新增数据集文件（建议）：

- `auto_test/datasets/probes/compression_retention_contract_v1.json`

字段建议：

1. `probe_id`
2. `retention_tier`
3. `memory_channel_expectation`（session_only/workbook_expected/files_expected）
4. `plant_text_template`
5. `probe_text_templates`
6. `judge_rubric`
7. `failure_policy`

## 3.4 统计口径（建议作为默认报表）

每次 run 输出以下核心表：

1. 会话级：
  - 每个 session 的 `must/should/may` 保留率
  - 压缩前后差分
  - 首次压缩后失败轮次
2. 聚合级：
  - `must_keep_post_compaction_failure_rate`
  - `should_keep_post_compaction_failure_rate`
  - `may_drop_retention_rate`（这个可低，不直接算错）
  - `false_alarm_rate`
3. 诊断级：
  - `cause_hint` 分布（compression_related vs task_switch 等）
  - 失败样本证据索引（turn refs）

## 3.5 更合理的“通过/失败”规则

建议最终 gating：

1. `must_keep`：任何压缩后确认失败，整 run 失败。
2. `should_keep`：失败率超过阈值（例如 >X%）判失败。
3. `may_drop`：不参与失败 gating，仅做趋势观测。

这比“单锚点连续两次 miss 就失败”更稳健。

## 4. 最终结论（直接回答你的核心问题）

你的判断是对的：当前机制确实存在方法论问题，根本原因不是“LLM判得准不准”，而是“被测对象定义错位”。

当前机制测到的是“锚点可用性波动”，不是严格意义上的“压缩失效边界”。

要把测试变成真正可用的压缩评估，必须同时做三件事：

1. 信息分层（must/should/may），替代“重要/不重要”二分法。
2. 压缩归因（pre/post compaction 对齐 + cause_hint）。
3. 统计升级（按层级和压缩后增量统计，而非只看首失效轮次）。

在你现有 `auto_test` 框架下，这套改造是可增量落地的，不需要推翻现有结果结构和脚本主干。
