# auto_test 递进开发方案（基于《测试.md》与当前实现）

## 1. 目标与边界

本方案目标：把 `auto_test` 从“单脚本可跑”升级为“可持续迭代的记忆/压缩评测平台”，并且严格对齐你在 [测试.md](/d:/04_projects/advoo_prod/ai-backend/auto_test/doc/测试.md) 中提出的三条主线：

1. 探针驱动（核心）
2. 静态基准（能力/回归分层）
3. Elo 对战（版本相对评估）

同时保留当前已跑通能力：

- 多轮会话自动执行（`run_5turn_session_test.py`）
- 模型扮演用户（`user_simulator`）
- workspace 导出（含 DotAI FS 回退）
- LLM 评估与报告落盘

---

## 2. 当前项目现状（作为起点）

已具备（可复用）：

- 测试执行主链路已稳定：
  - 创建会话 -> 多轮执行 -> SSE 解析 -> 结果落盘
- 结果结构已完善：
  - `results/session_autotest_<run_id>/workspace/` 有 `_manifest.json` 与文件导出
- 用户模拟已支持能力导向：
  - `capability_mode`（`alternating/mixed/copy_only/image_only/...`）
- 工具注册与交互回传已可控：
  - `option_card` 默认开，`post_submit/ui_cmd` 默认关，可环境变量切换
- 评估链路可用：
  - `dialogue_evaluator.py` + `llm_eval_system/user.prompt`

当前缺口（与《测试.md》对比）：

- 缺少“结构化探针数据集”和探针级评分器
- 缺少 pass@k / pass^k 的统计
- 缺少“能力基准 vs 回归基准”分层运行机制
- 缺少版本对战、Elo 计算与显著性判断
- 缺少稳定的数据治理（case 版本化、评分器回归验证）

---

## 3. 总体架构（目标态）

建议把评测拆成四层：

1. `Scenario Runner`：跑会话、采集轨迹（你现有代码为基础）
2. `Probe Engine`：按探针定义执行判定（确定性优先，LLM补充）
3. `Benchmark Layer`：能力集/回归集/静态集分开跑
4. `Arena Layer`：同题双版本对战 -> Elo + 统计显著性

---

## 4. 递进实施阶段

## Phase 1（先做）：探针体系 MVP（确定性优先）

目标：
- 建立可维护的探针定义格式与探针评分器

主要改造：
- 新增目录
  - `auto_test/datasets/probes/`
  - `auto_test/src/probe/`
- 新增数据结构（建议 JSON）：
  - `probe_id`
  - `probe_type`（`recall/artifact/continuation/decision`）
  - `target_turn` 或 `turn_window`
  - `judge_mode`（`deterministic` / `llm`）
  - `assertions`（`contains/regex/json_path/file_exists`）
- 在 `run_5turn_session_test.py` 后处理阶段接入 probe 执行器：
  - 读取 `turn_results.json` + `workspace/_manifest.json`
  - 输出 `non_text/probe_results.json`
  - 在 `evaluation.md` 增加 Probe 小节

验收标准：
- 至少 20 个探针可执行
- 其中 >=70% 使用确定性评分器（字符串、正则、路径断言）
- 生成 probe 级别通过率、失败原因、证据片段

---

## Phase 2：LLM 评分聚焦主观探针（延续/决策）

目标：
- 按《测试.md》修正项，LLM 仅用于主观探针，减少噪音和成本

主要改造：
- 新增 `auto_test/prompts/probe_llm_judge_*.prompt`
- 在 `src/eval/dialogue_evaluator.py` 中新增 probe 专用评审入口
- 维度固化为 6 维（0-5）：
  - Accuracy
  - Context awareness
  - Artifact trail
  - Completeness
  - Continuity
  - Instruction following
- 仅对 `continuation/decision` 探针走 LLM 评审

验收标准：
- 同一批次重复 3 次，LLM 维度波动可控（方差有记录）
- LLM 评分都能回溯到对应 turn 和证据
- 总评可拆解为“确定性分 + LLM主观分”

---

## Phase 3：pass@k / pass^k + 能力/回归双基准

目标：
- 把“能力上限”和“稳定性”分开度量，解决一次性跑分不可解释问题

主要改造：
- 新增批跑脚本：
  - `auto_test/src/tests/run_suite.py`（建议名）
- 新增基准分层：
  - `auto_test/datasets/benchmark_capability/`
  - `auto_test/datasets/benchmark_regression/`
- 统计指标：
  - `pass@k`：k 次任一成功
  - `pass^k`：k 次连续成功
- 报告增加分层汇总：
  - `results/<run>/non_text/suite_summary.json`

验收标准：
- 可对同一 case 一键跑 `k=5/10`
- 能力基准和回归基准分开出报告
- 回归基准可设置硬门槛（例如通过率接近 100%）

---

## Phase 4：静态数据集体系化（偏局部）

目标：
- 建立长期稳定的“事实真相”样本，支撑回归检测和上线门禁

主要改造：
- 新增静态集目录：
  - `auto_test/datasets/static/`
- 每个样本包含：
  - 输入上下文
  - 目标输出/拒答边界
  - 评分脚本或断言规则
- 增加数据版本字段：
  - `dataset_version`
  - `case_owner`
  - `last_validated_at`

验收标准：
- 静态集可被 suite runner 统一调度
- 每次改动可看到“新增失败 / 修复失败 / 稳定通过”
- 形成最小上线门禁规则（回归集未达阈值不通过）

---

## Phase 5：Elo 对战（偏整体）+ 显著性判断

目标：
- 支持版本 A/B 在同一动态用例下对战，输出可信排名

主要改造：
- 新增 Arena 执行器：
  - `auto_test/src/arena/run_pairwise_arena.py`
- 输入：
  - 两组配置（A/B）
  - 同一 case 列表
  - 同一角色种子/任务目标（保证可比性）
- 输出：
  - `arena_matches.jsonl`（逐场胜负）
  - `elo_ranking.json`（总榜 + 分榜）
  - `significance.json`（差异置信区间、标准误）

关键规则：
- 只有当“提升幅度 > 噪声区间（置信区间为正）”才判定有效提升
- 除总榜外保留分榜（防止单一标量掩盖短板）

验收标准：
- 同一版本重复对战排名稳定
- 出现循环胜负（A>B, B>C, C>A）时，报告中有维度解释
- 结果可直接用于版本发布决策

---

## Phase 6（预研）：Persona 与多目标优化

目标：
- 提升对话真实性与评估外推性，并探索质量-时延-成本平衡

内容：
- 引入更真实角色模拟框架（可先做轻量 persona 库）
- 引入“毒性/越权探针”作为安全侧补充
- 做多维帕累托分析：
  - 质量（probe/benchmark/elo）
  - 速度（端到端延迟）
  - 成本（token 与调用次数）

说明：
- 该阶段不建议阻塞前五阶段交付，作为并行预研更合理

---

## 5. 建议里程碑与产物

里程碑 M1（Phase 1-2）：
- 产物：探针引擎 + 混合评分（确定性+LLM）
- 可回答问题：压缩后“哪些信息丢了、丢在何处、影响哪类任务”

里程碑 M2（Phase 3-4）：
- 产物：套件批跑 + pass@k/pass^k + 能力/回归分层 + 静态集
- 可回答问题：模型是“偶尔聪明”还是“稳定可靠”

里程碑 M3（Phase 5）：
- 产物：Elo 对战系统 + 显著性判断
- 可回答问题：新版本是否“统计显著”优于旧版本

---

## 6. 与当前代码的最小侵入落地策略

优先遵循“加法改造”，避免破坏现有单脚本可用性：

1. 保留 `src/tests/run_5turn_session_test.py` 作为基础 runner
2. 新能力尽量以后处理方式接入（读取现有 `results` 产物）
3. 稳定契约只在必要时更新 [STABLE_INTERFACE_AND_RESULT_SPEC.md](/d:/04_projects/advoo_prod/ai-backend/auto_test/doc/STABLE_INTERFACE_AND_RESULT_SPEC.md)
4. 新增脚本/目录，不重命名现有关键入口

---

## 7. 风险与防偏策略

风险：
- LLM Judge 漂移导致评估不稳定
- Elo 单标量掩盖细分能力退化
- 网络抖动影响时延与成本统计

对应策略：
- 评分拆层：确定性优先，LLM 仅判主观项
- 总榜 + 分榜并行，附维度解释
- 性能统计引入去噪（多次采样、中位数/分位数）

---

## 8. 建议先后顺序（执行优先级）

1. Phase 1 探针引擎 MVP
2. Phase 2 主观探针 LLM 评审
3. Phase 3 pass@k/pass^k + 双基准分层
4. Phase 4 静态数据集治理
5. Phase 5 Elo + 显著性
6. Phase 6 Persona/安全/帕累托预研

这条顺序可以保证：每一阶段都能独立交付并直接提升你当前评估质量，不会出现“等大一统平台完成前都无法产出”的空档期。
