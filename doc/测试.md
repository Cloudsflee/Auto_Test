### 一、 记忆与压缩评估（以探针为核心的相对量化系统）

​    衡量“记忆有没有被过度压缩”时，一般存在两种路线：

- **被淘汰的方案：传统NLP字面指标（如 ROUGE、BLEU、精确匹配）** 这类指标只计算压缩后的文本与原文本在字词上的重叠度 。对于Agent来说这是无效的，因为模型可能会用完全不同的词汇总结出正确的语义，或者虽然字面重叠度很高，但恰恰漏掉了能让任务继续推进的最关键的“文件路径”。
- **当前选择的方案：**
  - 
  - #### LLM-as-a-Judge（大模型作为裁判，G-Eval框架）
  - LLM 作为裁判则用于评估更复杂的**知识流失和指令遵循**情况。当前主流的评估框架（如 DeepEval, TruLens, Ragas）都依赖这种模式。
  - **实施细节：** 通常会使用一个能力更强的大模型，配合评估 Prompt 来打分。同时，可以通过构造少量示例进行校准等方式，让评估更准确（参考：[https://www.langchain.com/articles/llm-as-a-judge](https://www.langchain.com/articles/llm-as-a-judge) [https://www.patronus.ai/llm-testing/llm-as-a-judge）](https://www.patronus.ai/llm-testing/llm-as-a-judge）)
    1. ####  探针驱动测试（Probe-Based Testing）（核心）
  - 这是一种动态的、基于问答的黑盒测试方法。
  - **核心逻辑：** 预先在测试数据集中埋入关键的“探针（Probe）”（例如特定变量名、用户的特定约束条件）。在记忆压缩后，向系统提出依赖这些探针才能回答的问题。
  - **测试方式：**
    - *输入探针问题：* “之前用户要求部署环境的端口号是多少？”
    - *验证标准：* 如果 Agent 能基于**压缩后的记忆**准确回答出“8080”，则说明该维度的记忆压缩是成功的。
  - **探针分类**：
    - **召回探针 (Recall)**：测试对特定事实（如早期的报错信息）的记忆保留 。
    - **制品探针 (Artifact)**：测试对环境状态（如修改过哪些文件）的追踪 。
    - **延续探针 (Continuation)**：测试在记忆截断后，智能体能否正确规划下一步行动 。
    - **决策探针 (Decision)**：测试对过往决策理由的保留程度，防止智能体忘记“为何这么做”而陷入死循环 。
  - **优势：** 直接测量记忆的**可用性，**强制要求压缩算法必须识别并保留对推进任务有关键逻辑价值的信息，之后使用 LLM 评判员从六个维度进行评估
  - **LLM核心维度（维度设计仅仅针对探针测试有效）（参考** **[https://factory.ai/news/evaluating-compression](https://factory.ai/news/evaluating-compression)）：**
    - 准确性 (Accuracy)：测试技术细节是否正确（如文件路径、函数名、错误信息等）。 
    - 上下文感知 (Context awareness)：测试回复是否反映了当前的对话状态。 
    - 制品追踪 (Artifact trail)：测试智能体是否清楚哪些文件被读取或修改过。 
    - 完整性 (Completeness)：测试回复是否解答了问题的所有部分。 
    - 延续性 (Continuity)：测试工作能否在不重新获取信息的情况下继续进行。 
    - 指令遵循 (Instruction following)：测试回复是否遵循了探测（probe）格式。
    - [理由](https://mlarkus325dppd90.usttp.larksuite.com/wiki/IIgJw2LXYidMQWkGLkAujYDBtAe)
  - **打分（以分数作为最终评测结果）：**每个维度均采用详细的评分标准，评分范围为 0-5 分。评分标准明确规定了每个标准中 0 分（“完全不及格”）、3 分（“基本合格，但存在一些小问题”）和 5 分（“优秀，没有任何问题”）的含义。

### 二、 静态基准数据集（偏局部） 与 Elo相对排名(基于对比的相对量化系统) （偏整体）

- **传统方案：静态基准测试（如 MMLU, GLUE, SWE-bench）** 通常给出明确的选择题或单轮任务，让模型去拿绝对分数（如满分100） 。
  - **适用场景：**对于具有明确任务目标且输入输出映射相对稳定的能力层（例如基础的“指令遵循能力”、简单的单轮信息抽取或合规性安全性检查），维护一批静态数据集是必要的 。数据集包含提示词、输入上下文、期望的正确输出甚至拒绝回答的边界案例，最终构成了衡量AI质量、检测回归风险以及在正式部署前验证迭代效果的“事实真相（Ground Truth）” 。
  - **缺点：**易受到数据污染，并且无法模拟现实中Agent在多轮交互中遇到打断、工具调用失败等动态情况。
  - **补充：**随着版本迭代，数据集要及时更新
  - 虽然这被称为静态数据集，但其内容生成往往借助AI辅助标注（LLM-assisted Labeling）来提效。通过让次级LLM进行N-shot学习（少样本学习）以生成初始答案，再人工进行校验与修正边界偏差，快速构建结构化测试用例 。
- **Elo 等级分系统 (参考****[https://www.emergentmind.com/topics/elo-based-auto-evaluation-metrics](https://www.emergentmind.com/topics/elo-based-auto-evaluation-metrics)**   **[https://arxiv.org/html/2601.08829v1](https://arxiv.org/html/2601.08829v1))**
  - **简介：**通过让两个不同版本的Agent在同一个动态测试用例下“对战”，由LLM裁判判定哪条执行轨迹更优 。因为Agent的业务场景没有绝对的“满分标准”，这种成对比较（Pairwise Comparison）能够容忍一定的噪音，并且在不依赖绝对完美答案的情况下，随着测试次数增加，自动收敛出非常稳定的版本能力排行榜。
  - **核心理论：**
    - 在智能体量化评测中，框架（如Chatbot Arena或内部的大型竞技场）将两个不同记忆版本的智能体实例暴露于同一测试用例下，并由**多个带有不同专家预设人设**的LLM裁判系统（甚至加入人类盲测反馈）来判定谁的压缩推理轨迹更优 。如果令智能体版本A与版本B当前的Elo评分分别为R_A和R_B，那么版本A在对决中的预期胜率可通过经典的逻辑斯蒂分布公式计算：

    - 当版本A击败版本B时，其评分上升的幅度取决于预期胜率：如果版本A本身评分远低于版本B，这场“爆冷”胜利将带来巨大的分数奖励；反之，若A本就远强于B，则分数仅有微小提升 。通过这种自动化的成对比较更新规则，系统可以在不需要绝对完美答案的场景下，针对极为开放和模糊的Agent任务，收敛出一个极其稳健、可解释的序列化能力排行榜 。

### 四、修正

1. 并非所有的探针都需要 LLM 来打分，部分直观的直接使用字符串匹配、正则表达式或断言测试会更精确
2. 将LLM评分的机制仅保留给主观性较强的“延续探针”和“决策探针”
3. 引入非确定性指标（pass@k 与 pass^k），在核心探针上采用 pass@k（测试 k 次只要成功一次就算该节点保留）来评估模型的能力上限，用 pass^k（连续成功）来评估稳定性
4. 为 Elo 对局引入成对分析，在同一套动态测试用例下，计算它们得分差异的标准误，只有提升幅度大于统计学上的噪声波动时（即置信区间为正），令迭代有效
5. 区分能力评估与回归评估，一部分是**能力基准**（针对长文本极端压缩等难题）（通过率可以很低），用于指引开发方向；另一部分是**回归基准**（基础指令遵循、常规状态维持），期望通过率接近 100%,避免模型退化
6. 能力基准得分过高时，要变成回归基准
7. 帕累托分析对“速度（延迟）”和“成本（Token消耗）”极其敏感，要想办法避免网络延迟等对其造成的影响🤔

一些思考：

1. 对于探针，如果使用确定性评分器，感觉必须隔一段时间对评分器的效果进行评估（人工/LLM）
2. Elo 系统可以对效果进行一定的排序，但是多维能力压缩成单一标量，可能会出现分高但不一定“好用”的情况（部分能力对于用户来说可能会显得并不那么重要，不同维度的分数权重占比得细细思量），并且在某些奇怪测试集里，说不定会有 A 赢 B、B 赢 C、C 赢 A 的循环，ELO除了总榜应该还要有分榜，避免这种情况的出现
3. 依旧是安全问题，感觉为了避免提示词注入等攻击，应该还要设置一些“毒性/越权探针”（暂时不是很重要，先放一边吧）
